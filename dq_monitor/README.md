## Быстрый старт

### 1. Установить Python-зависимости

```bash
git clone https://github.com/yourname/dq-monitor.git
cd dq-monitor
pip install -r requirements.txt
```

### 2. Убедиться, что готовые модули работают

```bash
pytest tests/
# Должно быть: 11 passed
```

### 3. Запустить приложение

```bash
streamlit run app/streamlit_app.py
# Открыть http://localhost:8501
```

Главная страница работает сразу — можно загрузить датасет и увидеть
сводку. Остальные страницы покажут «🔲 модуль ждёт реализации» — это
ваша работа.

### 4. Запустить через Docker (после реализации)

```bash
docker build -t dq-monitor .
docker run --rm -p 8501:8501 -v $(pwd)/data:/app/data dq-monitor
```

Не пугайтесь, если впервые видите Docker — см. `docs/DOCKER_FOR_BEGINNERS.md`,
там 15-минутный туториал.

## Архитектурные принципы

**1. `src/` не знает про Streamlit.**
Любая функция в `src/` принимает DataFrame и возвращает DataFrame
или dataclass. Streamlit только вызывает функции и отображает результат.
Это даёт чистое разделение бизнес-логики и UI — можно тестировать
юнит-тестами, переиспользовать вне приложения.

Если в вашем `src/*.py` появился `import streamlit` — это red flag.

**2. Данные между страницами — через `st.session_state`.**
Главная страница загружает CSV в `st.session_state["df_dirty"]`.
Дальше все страницы читают оттуда. Не передавайте через файлы и
не загружайте CSV повторно на каждой странице.

**3. Каждое правило — отдельная функция.**
И DQ-проверки в profiler, и правила очистки в cleaner, и антифрод-правила в antifraud_rules — все они должны быть независимыми функциями.
Так можно их тестировать по отдельности и включать/выключать в UI.
