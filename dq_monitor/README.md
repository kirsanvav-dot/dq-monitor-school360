# DQ Monitor

Каркас приложения для аудита качества данных в антифрод-системе банка.

**Команда:** 6 человек, 2 недели, школа ИИ360.

## Идея

Загружаете CSV с банковскими транзакциями → приложение находит DQ-проблемы по 4 измерениям → чистит данные → прогоняет rule-based антифрод-правила → показывает, насколько сильнее правила работают на чистых данных. Главный нарратив: **«грязные данные = пропущенный фрод»**.

## Что значит «контракт»

В файлах с 🔲 уже описаны:
- **Dataclass'ы** (`DQIssue`, `DQReport`, `CleaningConfig`, `RuleResult` и т.д.) — это структуры данных, которые ваш код возвращает. Они импортируются из других мест проекта, **не меняйте их имена и поля**.
- **Сигнатуры публичных методов** (`DataProfiler.profile`, `DataCleaner.clean`, `RuleEngine.run_all`, `compute_dq_score`) — это API, который вызывает Streamlit-приложение. **Не меняйте имена и типы аргументов/возвращаемых значений**, иначе UI сломается.

Всё остальное — на ваше усмотрение: внутренняя архитектура классов, приватные методы, имена переменных, организация кода внутри файла.

Сейчас публичные методы поднимают `NotImplementedError`. Когда вы их реализуете — соответствующая страница Streamlit оживёт.

## Данные

Ментор выдаёт два файла:

**День 1.** `data/raw/events_dirty.csv` (~150k строк) — это «грязные» данные банка. С ним работаете всю первую неделю.

**День 7-8.** `data/ground_truth/fraud_labels.csv` — эталонные метки фрода (`event_id` → `is_fraud_real`). Используете для финальной оценки на странице Antifraud Demo. До этого момента — работаете слепо, как настоящий аналитик.

⚠️ **Важно:** в датасете есть колонка `is_flagged`. **Это не ground truth.** Это пометки старой антифрод-системы банка, которая работает плохо. Подробнее — в карточке проекта.

## Архитектурные принципы

**1. `src/` ничего не знает про Streamlit.** Любая функция в `src/` принимает DataFrame и возвращает DataFrame или dataclass. Это даёт:
- модули можно тестировать юнит-тестами
- приложение можно переписать на любой другой фронтенд без переписывания логики
- код переиспользуем как библиотека

Если в вашей реализации в `src/` появился `import streamlit` — что-то пошло не так.

**2. Передача данных между страницами — через `st.session_state`.** Загруженный DataFrame, результаты профилирования, метрики — всё кладётся в session_state главной страницей или предыдущими страницами. Это нативный механизм Streamlit, не изобретайте альтернативы.

**3. Каждое правило — отдельная функция.** Это касается и DQ-правил в `profiler.py`, и антифрод-правил в `antifraud_rules.py`. Позволяет включать/выключать их по чекбоксу в UI и независимо тестировать.

**4. Все настройки — через dataclass'ы.** `CleaningConfig` — это конфиг очистки. Чистый Python-объект, легко сериализовать в JSON, легко передавать между функциями.
