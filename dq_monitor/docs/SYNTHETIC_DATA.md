# Synthetic Data — генерация тестовых датасетов

В системе есть встроенный генератор синтетических данных. С его помощью
вы можете создать несколько разных тестовых датасетов и показать на
защите, что DQ Monitor работает не только на одном CSV, а на любом
датасете, соответствующем `DATA_CONTRACT.md`.

## Зачем это нужно

Главный риск на защите — вопрос жюри:
> «А вы случайно не подогнали систему под конкретный CSV?»

Ответ — показать demo на разных датасетах: «вот low-severity → инструмент
находит мало проблем, вот high-severity → много проблем. Логика та же,
цифры разные, направление эффекта стабильное».

## Использование

### Запустить готовый скрипт

```bash
python scripts/generate_test_dataset.py
```

Он создаст три датасета: `low_severity/`, `medium_severity/`, `high_severity/`
в папке `data/test_datasets/`.

### Вызвать из своего кода

```python
from dq_monitor.src.synthetic_data import generate_events

df_dirty, df_clean, fraud_labels = generate_events(
    seed=42,
    n_transactions=50_000,
    n_clients=10_000,
    fraud_intensity="medium",
    dq_severity="medium",
)
```

`df_dirty` — то, что подаётся в инструмент на главной странице
(без колонки `is_fraud_real`, как реальные данные банка).
`df_clean` — эталон с полной разметкой, для проверки своей логики.
`fraud_labels` — таблица `event_id → is_fraud_real`, нужна для расчёта
confusion matrix на странице Antifraud Demo.

### Сохранить в файлы

Передайте `output_dir`:

```python
generate_events(
    seed=123,
    output_dir="data/test_datasets/my_custom",
)
```

Это создаст:
```
data/test_datasets/my_custom/
├── raw/
│   └── events_dirty.csv
└── ground_truth/
    ├── events_clean.csv
    └── fraud_labels.csv
```

## Параметры

| Параметр | Тип | По умолчанию | Что значит |
|----------|-----|--------------|------------|
| `seed` | int | 42 | Зерно генератора. Одинаковый seed → идентичный датасет |
| `n_transactions` | int | 50_000 | Сколько нормальных событий |
| `n_clients` | int | 10_000 | Сколько уникальных клиентов |
| `fraud_intensity` | str | "medium" | `low` / `medium` / `high` |
| `dq_severity` | str | "medium" | `none` / `low` / `medium` / `high` |
| `output_dir` | str/Path | None | Куда сохранять CSV. None — только возврат |

### Пресеты

**`fraud_intensity`** управляет тем, сколько фрода в данных:
- `low` — половина от medium, удобно для baseline
- `medium` — реалистичный уровень (~1.6% строк)
- `high` — больше фрода, нагрузочный режим

**`dq_severity`** управляет тем, сколько DQ-проблем:
- `none` — чистые данные без закладок. **Полезно**: подайте такой
  датасет в DQ Monitor — система должна находить почти ничего.
  Если находит много — есть false positive в детекторах
- `low` — 30% от medium
- `medium` — реалистичный уровень
- `high` — много мусора, нагрузочный тест очистки

## Что внутри сгенерированного датасета

Те же 20 колонок, что и в `DATA_CONTRACT.md`. DQ-проблемы покрывают
большинство категорий из `constant_issue.py`:

- Completeness — пропуски в обязательных полях (event_id, client_id,
  event_type, event_ts, device_type, ip_address, geo_country, geo_city,
  channel) и в transaction/session-специфичных полях
- Validity — битые форматы дат, IP, currency, MCC-коды вместо
  текстовых категорий, аномальные суммы, невалидные device_type/channel,
  кривой card_last4
- Consistency — рассогласование is_flagged/flag_reason, перемешанные
  поля transaction/session, session_end раньше session_start
- Uniqueness — полные дубликаты строк, дублирующиеся event_id

Точные пропорции каждой проблемы — закрытая часть генератора. Если бы
вы их знали, проверки в `profiler.py` могли бы быть «подогнаны под
эти числа», и тест перестал бы быть честным.

## Сценарий для защиты

Хороший вариант финального демо:

1. **Главная демонстрация** — на `data/raw/events_dirty.csv` (тот, что
   изначально получили от ментора). Показываете весь пайплайн
   и главную метрику «+N фродов после очистки».

2. **Стресс-тест** — переключаетесь на `data/test_datasets/high_severity/`
   и запускаете тот же пайплайн. «Смотрите, на других данных система
   находит другие проблемы, но архитектура работает одинаково».

3. **Baseline** — `data/test_datasets/none/` (с dq_severity="none").
   Показывает, что система не ложно срабатывает на чистых данных:
   DQ-score близок к 1.0, проблемы почти не находятся.

Это сильно. Жюри не сможет сказать «вы подогнали под один файл».

## Часто задаваемые вопросы

**Можно ли использовать сгенерированный датасет вместо исходного?**
Можно для отладки и тестов, но **основная демо-защита должна быть
на CSV от ментора**. Он эталонный — ментор знает, какие именно
закладки в нём, и сможет проверить корректность ваших находок.

**Точно ли сгенерированный датасет соответствует контракту?**
Да. Все 20 колонок, правильные типы. Если ваш `data_loader.py`
работает с CSV от ментора — будет работать и с этим.

**Сколько времени занимает генерация?**
- 5 000 строк: ~1 секунда
- 50 000 строк: ~5-10 секунд
- 150 000 строк: ~30 секунд

**Я задал seed=42, но получил другой результат. Почему?**
Скорее всего, изменили другие параметры (`n_transactions`,
`fraud_intensity`). Воспроизводимость гарантируется только при
**идентичном наборе** всех параметров.

**Можно ли поменять конкретные значения опечаток в currency?**
Нет. Это часть учебной задачи — вы должны открыть данные,
посмотреть на `value_counts()` и принять решение сами. Если генератор
давал бы вам готовый словарь, проект терял бы смысл.

**А если я просто открою исходник `synthetic_data.py` и посмотрю?**
Технически можете. Но имейте в виду: на защите жюри спросит, как
вы пришли к своему `CURRENCY_MAPPING`. Если ответ «подсмотрели в
генераторе» — это слабо. Если «увидели в `df['currency'].value_counts()`,
обсудили в команде и зафиксировали в `reference_data.py`» — сильно.
