# Docker для нашего проекта — 15-минутный туториал

Эта инструкция для тех, кто **слышал слово Docker, но никогда не использовал**.
К концу туториала у вас на ноутбуке будет запущенное в контейнере
Streamlit-приложение проекта.

## Зачем нам Docker

Простой ответ: на защите вам нужно показать приложение жюри. Если запускать
его «как обычно», то на новом компьютере придётся:
1. Установить правильную версию Python
2. Установить все библиотеки из requirements.txt
3. Надеяться, что ничего не конфликтует с тем, что уже стоит
4. Запустить приложение

Это работает, пока вы запускаете на своём ноутбуке. На чужом — ломается.

С Docker всё иначе: вы один раз описываете в `Dockerfile` рецепт сборки,
получаете «образ» — это полностью изолированная среда с Python и нужными
библиотеками. На любом компьютере с установленным Docker одна команда
`docker run` запускает приложение. Никаких конфликтов, никаких сюрпризов.

В реальной компании это стандартный способ деплоить приложения — поэтому
вы заодно поучите индустриальный навык.

## Базовые понятия (1 минута)

- **Образ (image)** — это «снимок» приложения с операционной системой
  и всеми зависимостями. Что-то вроде ISO-файла или шаблона виртуальной
  машины. Образ статичен, не работает.
- **Контейнер (container)** — это запущенный экземпляр образа. Запускается,
  работает, останавливается, удаляется. Можно одновременно запустить
  несколько контейнеров из одного образа.
- **Dockerfile** — текстовый файл с инструкциями: «возьми Python 3.11,
  скопируй сюда мой код, поставь библиотеки, запусти такую-то команду».
  Из Dockerfile собирается образ.

Аналогия: Dockerfile = рецепт, образ = торт по рецепту, контейнер = кусок,
который вы едите. Один рецепт → много тортов → ещё больше кусков.

## Установка Docker (5 минут)

### Windows / macOS

1. Скачать **Docker Desktop** с [docker.com/products/docker-desktop](https://www.docker.com/products/docker-desktop)
2. Установить с настройками по умолчанию
3. После установки запустить Docker Desktop из приложений
4. В правом нижнем углу (Windows) или в строке меню (Mac) появится иконка
   с китом. Когда она перестанет анимироваться — Docker готов
5. Открыть терминал (на Windows — PowerShell или git-bash) и проверить:
   ```
   docker --version
   ```
   Должно появиться что-то вроде `Docker version 26.1.4, build...`

⚠️ **Важно для Windows:** если Docker просит включить WSL 2 — соглашайтесь
и следуйте инструкции. Это бесплатно, безопасно и нужно для работы Docker
на Windows.

⚠️ **Важно для Mac:** Docker Desktop требует ~3 ГБ места и заметно ест RAM
(2-4 ГБ). На макбуках с 8 ГБ может быть тесновато — закройте Chrome и
тяжёлые приложения перед запуском.

### Linux

```bash
# Ubuntu/Debian
sudo apt update
sudo apt install docker.io
sudo usermod -aG docker $USER   # чтобы запускать без sudo
# Выйти из сессии и зайти обратно

docker --version
```

## Наш Dockerfile разбор (3 минуты)

Откройте `Dockerfile` в корне проекта — он короткий и закомментирован.
Прочитайте сверху вниз:

```dockerfile
FROM python:3.11-slim
```
Берём готовый базовый образ — Linux с уже установленным Python 3.11.
`slim` означает «без лишнего», ~50 МБ вместо ~900 МБ.

```dockerfile
WORKDIR /app
```
Внутри контейнера переходим в директорию `/app`. Дальше все команды
работают там.

```dockerfile
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
```
Копируем только `requirements.txt` и ставим зависимости. **Делаем это
ДО копирования кода** — это трюк ради кеша: пока зависимости не меняются,
этот шаг переиспользуется из кеша, и каждая сборка после правки кода
занимает не 2 минуты, а 5 секунд.

```dockerfile
COPY src/ ./src/
COPY app/ ./app/
```
Копируем код. Если код изменился — пересоберётся только этот слой.

```dockerfile
EXPOSE 8501
CMD ["streamlit", "run", "app/streamlit_app.py", \
     "--server.address=0.0.0.0", \
     "--server.port=8501", \
     "--server.headless=true"]
```
`EXPOSE` — документация о том, что приложение слушает порт 8501.
`CMD` — что запустить, когда контейнер стартует. Параметры Streamlit:
- `--server.address=0.0.0.0` — слушать на всех интерфейсах (иначе изнутри
  контейнера приложение будет недоступно снаружи)
- `--server.headless=true` — не пытаться открыть браузер (его в контейнере нет)

## Сборка и запуск (3 минуты)

В корне проекта (там, где Dockerfile):

### Шаг 1: собрать образ
```bash
docker build -t dq-monitor .
```
Разбор:
- `docker build` — команда сборки образа
- `-t dq-monitor` — назвать образ `dq-monitor` (tag = метка)
- `.` — путь к директории с Dockerfile (текущая)

Первый раз сборка займёт 2–5 минут (скачивается базовый образ Python и
ставятся библиотеки). Последующие сборки — 5–10 секунд, если меняли
только код.

В конце увидите что-то вроде `Successfully tagged dq-monitor:latest`.

### Шаг 2: запустить контейнер
```bash
docker run --rm -p 8501:8501 -v $(pwd)/data:/app/data dq-monitor
```
Разбор:
- `docker run` — запустить контейнер
- `--rm` — удалить контейнер после остановки (чтобы они не накапливались)
- `-p 8501:8501` — пробросить порт: `8501 на вашем компьютере` → `8501 в контейнере`
- `-v $(pwd)/data:/app/data` — пробросить папку: `./data на вашем компьютере` ↔
  `/app/data в контейнере`. Так контейнер видит ваш `events_dirty.csv`.
- `dq-monitor` — имя образа

⚠️ **На Windows вместо `$(pwd)/data`** используйте `%cd%/data` (CMD) или
`${PWD}/data` (PowerShell), либо абсолютный путь.

Streamlit запустится. Откройте в браузере **http://localhost:8501** — должно
открыться то же приложение, что и при локальном запуске.

### Шаг 3: остановить контейнер

В терминале, где запущен docker run, нажмите `Ctrl+C`. Контейнер
остановится и удалится (потому что был флаг `--rm`).

## Makefile-шорткаты

Чтобы каждый раз не набирать длинные команды, в проекте есть `Makefile`:

```bash
make docker-build      # вместо docker build -t dq-monitor .
make docker-run        # вместо docker run --rm -p 8501:8501 ... dq-monitor
```

⚠️ **На Windows `make` нет «из коробки».** Варианты:
- Если вы установили Git Bash + WSL — make там обычно есть
- Поставить отдельно: `winget install GnuWin32.Make` или через Chocolatey
- Или просто копировать команды из Makefile вручную — там нет магии

## Частые проблемы

**`docker: command not found`** — Docker не установлен или Docker Desktop
не запущен. Откройте приложение Docker Desktop.

**`Cannot connect to the Docker daemon`** — на Mac/Windows запустите
Docker Desktop. На Linux: `sudo systemctl start docker`.

**`port is already allocated`** — на порту 8501 уже что-то крутится.
Либо остановите старый процесс (Streamlit запущен в другом терминале?),
либо запустите на другом порту: `-p 8502:8501`, потом откройте
http://localhost:8502.

**`No space left on device`** — Docker накопил мусор. Чистка:
```bash
docker system prune -a
```
Удалит все неиспользуемые образы и контейнеры. **Внимание:** все ваши
образы перестроятся при следующем `docker build`.

**Сборка медленная даже на повторных вызовах** — вы редактируете
`requirements.txt` или сам Dockerfile. Слой с зависимостями невалидируется,
и pip install запускается заново. Это нормально.

**Внутри контейнера не видно `events_dirty.csv`** — забыли флаг `-v`,
который пробрасывает папку data. Перепроверьте команду `docker run`.

**На Mac M1/M2 какая-то ошибка про архитектуру** — добавьте флаг при сборке:
```bash
docker build --platform linux/amd64 -t dq-monitor .
```

## Что показать на защите

На защите вы продемонстрируете жюри две вещи:

1. **Запуск через Docker:**
   ```bash
   docker run --rm -p 8501:8501 -v $(pwd)/data:/app/data dq-monitor
   ```
   И откроете приложение в браузере. Жюри увидит, что у вас не «скрипт в
   ноутбуке», а упакованное продакшн-ready приложение.

2. **Способность ответить на вопрос «как кто-то ещё это запустит»:**
   «Клонирует репозиторий, ставит Docker, запускает `docker build` и
   `docker run`. Всё. Никаких «у меня pandas другой версии стоит».

Этого достаточно для защиты. Глубже Docker копать не нужно — orchestration
с docker-compose, registry, multi-stage builds — это уже для других курсов.

## Если совсем не получается

Не паникуйте: Docker — это **бонусная функциональность**, не критичная
для проекта. Если за 30 минут не удалось установить или запустить:
- Запишите проблему в чат команды
- Покажите приложение на защите без Docker — через `streamlit run`
- Скажите жюри честно: «Dockerfile в репозитории есть, но запустить
  не удалось — это наша точка роста». Это нормальный ответ.

Главная ценность проекта — DQ Monitor и его выводы, а не контейнеризация.
